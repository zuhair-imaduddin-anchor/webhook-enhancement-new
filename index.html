Future State: Transfer Workflow

With Transfer ID included in the webhook payload, enabling real-time correlation.

Client Application

1. Receive Webhook: transfer.initiated
Action: Parse payload. Extract both transactionId and transferId immediately.
2. State Update: Update internal DB.
Action: (Optional) Send GET /v2/transfers/{transferId} if extra metadata is needed.
3. Receive Webhook 2: transfer.completed
Action: Parse payload. Extract transferId.
4. Complete Workflow:
Action: Mark transfer as complete in local system. Zero reconciliation required.

Backend API

1. Fire Webhook:
Action: Send updated payload containing transactionId AND transferId.
2. Process Request (Light):
Action: If client requests metadata, serve a highly efficient, indexed point-lookup by ID.
3. Fire Webhook:
Action: Send updated payload containing transactionId AND transferId.
4. No Action Required:
Action: Backend is completely free from unnecessary polling or heavy list queries.

Impact Metrics (Per 1,000 Transfers)

  • Total API Requests: 0 to 2,000 max (Down from 4,000). Clients can rely fully on webhooks or do 1 direct lookup.
  • Rate Limiting Impact: MINIMAL. 100% reduction in heavy GET /v2/transfers queries. Massive improvement in database performance and client UX.
